# v1.5 migration guide (v1 ➜ v2 bridge)

v1.5 ships a transitional base class (`BasePageV1toV2`) that ingests v1 `addSchema` definitions while exposing only v2
registry accessors. The bridge is **deprecated** and will be removed in 2.0.0; 1.5.0 is the final v1 release. Prefer moving
directly to the v2 base (`PageObject`) whenever possible.

## What changes in v1.5

- `BasePage` (v1) remains available and behaves as before, so upgrading to v1.5.0 won't affect anything before you perform migration changes.
- `BasePageV1toV2` translates `addSchema(...)` calls into the v2 registry but only exposes the v2 accessors
  (`add`, `getLocator`, `getNestedLocator`, `getLocatorSchema`).
- `initLocatorSchemas` is still invoked for v1 definitions and used to populate the v2 registry unless a path as already been migrated to defineLocators, initLocatorSchemas is marked deprecated and warns at runtime.
- The v1→v2 translator logs when a schema cannot be translated (e.g., missing fields or Locator instances) so you can rewrite
  those paths in `defineLocators` early.

## Preferred migration path (single step)

If your suite can tolerate updating call sites and locator definitions together, skip the bridge and use `PageObject`.

```ts
// before (v1)
import { BasePage } from "pomwright";
class Login extends BasePage<LocatorSchemaPath> { /* ... */ }

// after (v2)
import { PageObject } from "pomwright";
class Login extends PageObject<LocatorSchemaPath> {
  protected defineLocators(): void {
    this.add("main").locator("main");
    this.add("main.button@login").getByRole("button", { name: "Login" });
  }
}
```

Checklist for the single-step switch:

- Extend `PageObject`.
- Move locator definitions into `defineLocators` using the v2 DSL.
- Remove `initLocatorSchemas` implementations when they have been migrated to defineLocators.
- Update no longer supported v1 syntax for methods `getLocator`, `getNestedLocator` and `getLocatorSchema` to the new v2 syntax in tests/helpers/POCs/fixtures/etc.

## Phased migration using the bridge

Use this when you need a staged rollout across layered base classes.

1. **Switch inheritance to `BasePageV1toV2`.** Fix call-site syntax to v2 accessors; v1 methods are no longer exposed.
2. **Keep `initLocatorSchemas` temporarily** for existing `addSchema` calls; the bridge translates what it can into the v2
   registry and logs any skipped paths so they can be fixed.
3. **Incrementally port definitions to `defineLocators`.** Remove the translated `addSchema` entry as you rewrite each path. Drop `initLocatorSchemas` entirely once all paths live in `defineLocators`.
4. **Finish by extending `PageObject`.** Then update POCs accordingly.

### Bridge-specific notes

- Runtime warnings fire when `BasePageV1toV2` is constructed and when `initLocatorSchemas` is invoked per POC, reminding you of the removal in
  2.0.0.
- Translator gaps: schemas with missing selector fields or `Locator` instances are skipped and logged; rewrite those in
  `defineLocators` immediately.
- The bridge narrows `this.locators` to `addSchema` only to discourage new v1-style mutations.

## Example: bridge with translated v1 schemas

```ts
import type { Page, TestInfo } from "@playwright/test";
import type { PlaywrightReportLogger } from "pomwright";
import { BasePageV1toV2, GetByMethod } from "pomwright";

type LocatorSchemaPath = "main" | "main.button@login";

export default class Login extends BasePageV1toV2<LocatorSchemaPath> {
  constructor(page: Page, testInfo: TestInfo, pwrl: PlaywrightReportLogger) {
    super(page, testInfo, "https://example.com", "/login", Login.name, pwrl);
  }

  protected defineLocators(): void { // v2
    this.add("main").locator("main");
    this.add("main.button@login").getByRole("button", { name: "Login" });
  }

  protected initLocatorSchemas(): void { // v1 (deprecated, translated)
    this.locators.addSchema("main", {
      locator: "main",
      locatorMethod: GetByMethod.locator,
    });

    this.locators.addSchema("main.button@login", {
      role: "button",
      roleOptions: { name: "Login" },
      locatorMethod: GetByMethod.role,
    });
  }
}
```

## Translator limitations (action required)

- Definitions that rely on Playwright `Locator` instances cannot be translated; rewrite them in `defineLocators`.
- Missing selector/text/role fields are skipped; fix the v1 schema or rewrite directly in v2.
- Paths already registered in v2 (defineLocators) are ignored when initLocatorSchema is invoked and the translation of v1 LocatorSchema are used to populate the v2 registry.

## Direct migration example (no bridge)

```ts
// before: bridge with v1 definitions
class AccountPage extends BasePageV1toV2<LocatorSchemaPath> {
  protected initLocatorSchemas(): void {
    this.locators.addSchema("nav.settings", {
      locator: "nav.settings",
      locatorMethod: GetByMethod.locator,
    });
  }

  protected defineLocators(): void {
    this.add("nav.settings").locator("nav.settings");
  }
}

// after: final v2 form
import { PageObject } from "pomwright";
class AccountPage extends PageObject<LocatorSchemaPath> {
  protected defineLocators(): void {
    this.add("nav.settings").locator("nav.settings");
  }
}
```

## Choosing the right path

- Default recommendation: adopt `PageObject` directly for greenfield or low-coupling suites.
- Use the bridge only when layered base classes or widespread helpers make a single-step switch risky.
- Plan to remove `initLocatorSchemas` and stop extending `BasePageV1toV2` before upgrading to v2.0.0.
