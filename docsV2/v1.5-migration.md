# v1.5 migration guide (v1 ➜ v2 bridge)

v1.5 introduces a transitional base class (`BasePageV1toV2`) that preserves v1 behavior while exposing the v2 locator
registry and accessors. This allows incremental migration from v1 locator schemas to the v2 registry DSL before
adopting v2.0.0.

## What changes in v1.5

- v1 `BasePage` remains available and behaves as before.
- A new `BasePageV1toV2` class adds v2 registry accessors alongside the v1 APIs.
- v1 `addSchema(...)` calls are mirrored into the v2 registry through an internal translator. If a v1 path has not been
  registered via `registry.add(...)`, the bridge logs an informational message and auto-registers the definition.

## Recommended migration flow

1. **Switch POCs extending `BasePage` to extend `BasePageV1toV2`.**
2. **Add `protected defineLocators(): void` to POCs**
3. **Add `defineLocators(registry)` to locator schema files** and register v2 locators with `registry.add(...)`.
4. **Keep `initLocatorSchemas()` intact** so existing v1 locators remain available while you migrate.
5. **Update tests gradually** to use `getLocatorV2`, `getLocatorSchemaV2`, and `getNestedLocatorV2`.

## Example: updating a POC containing LocatorSchemaPath union type

```ts
import { BasePageV1toV2, GetByMethod } from "pomwright";
import type { PlaywrightReportLogger } from "pomwright";
import type { Page, TestInfo } from "@playwright/test";

type LocatorSchemaPath =
  | "main"
  | "main.button@login";

export default class Login extends BasePageV1toV2<LocatorSchemaPath> {
  constructor(page: Page, testInfo: TestInfo, pwrl: PlaywrightReportLogger) {
    super(page, testInfo, "https://example.com", "/login", Login.name, pwrl);
  }

  protected defineLocators(): void { // V2
    const { add } = this.locatorRegistry;
    add("main").locator("main");
    add("main.button@login").getByRole("button", { name: "Login" });
  }

  protected initLocatorSchemas(): void { // V1
    this.locators.addSchema("main", {
      locator: "main",
      locatorMethod: GetByMethod.locator,
    });

    this.locators.addSchema("main.form.button@login", {
      role: "button",
      roleOptions: { name: "Login" },
      locatorMethod: GetByMethod.role,
    });
  }
}
```

## Example: updating a POC using a locatorSchema file to the bridge base class

```ts
import type { Page, TestInfo } from "@playwright/test";
import type { PlaywrightReportLogger } from "pomwright";
import { BasePageV1toV2 } from "pomwright";
import { defineLocators, initLocatorSchemas, type LocatorSchemaPath } from "./login.locatorSchema";

export class LoginPage extends BasePageV1toV2<LocatorSchemaPath> {
  constructor(page: Page, testInfo: TestInfo, pwrl: PlaywrightReportLogger) {
    super(page, testInfo, "https://example.com", "/login", LoginPage.name, pwrl);
  }

  protected defineLocators(): void {
    defineLocators(this.locatorRegistry);
  }

  protected initLocatorSchemas(): void {
    initLocatorSchemas(this.locators);
  }
}
```

## Example: updating a locator schema file

```ts
import { GetByMethod, type GetLocatorBase, type LocatorRegistry } from "pomwright";

export type LocatorSchemaPath = "main" | "main.button@login";

export function defineLocators(registry: LocatorRegistry<LocatorSchemaPath>) { // V2
  const { add } = registry;
  add("main").locator("main");
  add("main.button@login").getByRole("button", { name: "Login" });
}

export function initLocatorSchemas(locators: GetLocatorBase<LocatorSchemaPath>) { // V1
  locators.addSchema("main", {
    locator: "main",
    locatorMethod: GetByMethod.locator,
  });

  locators.addSchema("main.button@login", {
    role: "button",
    roleOptions: { name: "Login" },
    locatorMethod: GetByMethod.role,
  });
}
```

## Example: migrating a test incrementally

```ts
// v1
const submit = await page.getNestedLocator("main.button@login");
await submit.click();

// v2 (new accessors)
await submit = page.getNestedLocatorV2("main.button@login").click();
```

## Tips

- If you see console messages about paths not yet migrated, add them to `defineLocators(...)` with `registry.add(...)`.
- Keep `LocatorSchemaPath` unions shared across v1 and v2 definitions to ensure consistent typing.
- Once all locators and tests use v2 APIs, you can migrate to `BasePageV2` and remove v1 schemas entirely.
